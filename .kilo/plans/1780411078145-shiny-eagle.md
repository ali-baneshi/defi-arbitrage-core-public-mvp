## 1. System Understanding Summary

**defi-arbitrage-core** is a boundary-scoped, offline DeFi market analysis kernel. It ingests local JSON MarketSnapshot files, validates them through a fail-closed multi-layer validation stack, runs a bounded cycle-detection engine, and emits deterministic Text/JSON opportunity reports. It does not touch networks, wallets, signing, RPC, or on-chain execution.

### Technology Stack
- **Python 3.10+** — canonical implementation, zero hard runtime dependencies
- **Rust 2021** (optional) — process-boundary analyzer for acceleration, not canonical
- **JSON schemas (draft 2020-12)** — public input/output contracts
- **Makefile + Docker + GitHub Actions** — build/CI/CD
- **pytest, ruff, cargo test** — test/lint layer

### Canonical Source Tree
| Module | Role |
|--------|------|
| `src/arbcore/models.py` | Frozen dataclasses: Edge, MarketSnapshot, Opportunity, RiskPolicy |
| `src/arbcore/validation.py` | Dependency-free payload shape checks |
| `src/arbcore/providers.py` | `JsonFileProvider` (local file only) |
| `src/arbcore/engine.py` | Bounded DFS cycle detector |
| `src/arbcore/reporters.py` | TextReporter / JsonReporter |
| `src/arbcore/service.py` | `RustAnalysisService` subprocess adapter + fallback |
| `src/arbcore/contracts/*` | Manifest loader + static Solidity/Vyper template validator |
| `src/arbore/cli.py` | CLI entrypoint |
| `schemas/*.json` | Public JSON contracts for inputs/outputs/diagnostics |

### Data Flow
```
Local JSON File
  → JsonFileProvider.load_snapshot()
  → validate_snapshot_payload()  (reject unknown keys, enforce sizes)
  → Edge/MarketSnapshot.normalized() + validate()
  → AnalysisEngine.analyze()    (DFS bounded by max_hops, policy-filtered)
  → Reporter.render()           (text or JSON)
```

### Invariants (documented & enforced)
- Offline only, no network required for baseline validation
- Fail-closed: reject unknown keys, reject unbounded settings, finite-only numerics
- Python is canonical; Rust is optional and additive
- Opportunity outputs are deterministic given the same policy and snapshot
- Contract templates are inert/`NOT AUDITED`, never compiled or deployed

## 2. Categorized, Prioritized Gaps & Hardening Needs

### CRITICAL

#### C-1. Historical credential exposure remains unverified
- **Location**: SECURITY.md, docs/en/HISTORY_CLEANUP.md, docs/en/RELEASE_EVIDENCE.md
- **Evidence**: `docs/en/HISTORY_CLEANUP.md:14` states "credentials that appeared in the previous local history must be considered compromised." `SECURITY.md:37` makes the same warning. `scripts/release_readiness.py:14-33` blocks public release pending `independent_history_scan` manual gate.
- **Impact**: Any leaked credentials (RPC URLs, API keys, wallet seeds) in the old `.git` history become discoverable by anyone who obtains a clone or mirror of a published repository.
- **Severity**: Critical
- **Likelihood**: Cannot confirm locally; must be resolved before publication

#### C-2. Absolute local path exposed via diagnostics
- **Location**: `src/arbcore/diagnostics.py:31`
- **Evidence**: `collect_diagnostics()` serializes `Path.cwd()` via `str(Path.cwd())` which is an absolute path. While `docs/en/RUNBOOK.md:101` says this is safe to paste, the path leaks hostname, username, and directory structure to anyone who receives a bug report or support request.
- **Impact**: Operational security / privacy leak of environment details.
- **Severity**: Critical (privacy)
- **Likelihood**: Certain if `--diagnostics` output is ever shared

### HIGH

#### H-1. Rust engine lacks metadata depth/key/byte limits (parity gap)
- **Location**: `rust/arbcore-rs/src/main.rs:256` vs `src/arbcore/models.py:86-98`
- **Evidence**: Python validates `MAX_METADATA_KEYS (128)`, `MAX_METADATA_BYTES (1,048,576)`, `MAX_METADATA_DEPTH (10)`. Rust calls `normalize_edge()` which only checks `normalized.metadata.is_object()` and accepts any JSON object without size or depth constraints.
- **Impact**: A snapshot rejected by Python would be accepted by Rust, breaking the "Python is canonical" invariant and creating divergent behavior between engines.
- **Severity**: High
- **Likelihood**: Certain if Rust falls back for large-metadata inputs

#### H-2. Rust response parsing does not cross-validate profit_bps against gross_return
- **Location**: `src/arbcore/service.py:171-177`
- **Evidence**: `_opportunity_from_dict()` accepts `profit_bps` as a raw float without verifying `profit_bps ≈ (gross_return - 1.0) * 10_000.0`. Python `Opportunity` is also not re-derived from the raw dict.
- **Impact**: A buggy or compromised Rust binary could return inconsistent numeric fields that pass validation. Correlates with H-1's pattern of trusting Rust output beyond structural shape.
- **Severity**: High (trust boundary violation)
- **Likelihood**: Low intrinsic but high impact; Rust output is untrusted per `docs/en/TRUST_BOUNDARIES.md:8`

#### H-3. Star-import obscures public API surface
- **Location**: `src/defi_arbitrage_core/__init__.py:3` and `src/defi_arbitrage_core/providers.py`, `reporters.py`
- **Evidence**: `from arbcore import *` re-exports everything. `__all__` in `arbcore.__init__` helps, but the pattern makes it hard to add vetting, audit, or documentation discipline, and could inadvertently expose internal symbols.
- **Impact**: Maintenance risk; accidental API leakage; refactoring risk.
- **Severity**: Medium-High
- **Likelihood**: Certain

#### H-4. Python analysis engine has no timeout or resource limit for large input graphs
- **Location**: `src/arbcore/engine.py:31-69`
- **Evidence**: DFS recursion in `_walk` is bounded only by `max_hops` (default 4, max 8) and node degree (max 500). For graphs with 10,000 edges and max degree at many nodes, combinatorial explosion is bounded in depth but not in branching. There is no `resource.setrlimit`, no `signal.alarm`, and no timeout.
- **Impact**: A crafted snapshot with high-degree nodes could cause long-running analysis loops (though bounded by hop depth, the combination still risks minutes of CPU for 8-hop × 500-degree × N-start scenarios).
- **Severity**: High (reliability/DoS)
- **Likelihood**: Moderate with adversarial/accidental large-degree inputs

#### H-5. Silent discard of resource limit setup errors
- **Location**: `src/arbcore/service.py:96-105`
- **Evidence**: `_pre_exec_resource_limit()` catches `(ImportError, ValueError, OSError)` and passes silently. On Windows, `resource` module is unavailable; the Rust subprocess would be launched without any memory limit.
- **Impact**: The single defense against untrusted Rust binary exhausting memory is silently disabled on Windows and on any platform where `setrlimit` fails.
- **Severity**: High
- **Likelihood**: Certain on Windows; low elsewhere

### MEDIUM

#### M-1. CLI silently falls back to `examples/market_snapshot.json` when no positional arg and ARBCORE_PROVIDER_FILE is unset
- **Location**: `src/arbcore/cli.py:81`
- **Evidence**: `snapshot_path = args.snapshot or settings.provider_file` where `settings.provider_file` defaults to `"examples/market_snapshot.json"`. A user running `defi-arbitrage-core --json` without arguments will silently analyze the example file.
- **Impact**: Automation masking (CI pipelines, scripts, user mistakes) could analyze stale example data instead of the intended snapshot.
- **Severity**: Medium
- **Likelihood**: High in automation contexts

#### M-2. Float serialization is not deterministic for diagnostics golden fixtures
- **Location**: `src/arbcore/diagnostics.py:51` (serializes `settings.policy.__dict__`); `src/arbore/reporters.py:20-21` (JsonReporter uses `json.dumps`); `src/arbcore/models.py:97` (`profit_bps=(gross_return - 1.0) * 10_000.0`)
- **Evidence**: Policy floats (e.g., `min_profit_bps=5.0`, `max_notional=10000.0`, `estimated_capacity`) are serialized by Python's default `float.__repr__`. Although CPython typically emits `10000.0` (not `1e4`), different platforms or minor Python patch versions can produce different string representations. `fixtures/golden_outputs/diagnostics.json` would break cross-platform.
- **Impact**: Cross-platform golden fixture validation; release evidence becomes platform-dependent.
- **Severity**: Medium
- **Likelihood**: Low on most Linux, higher on Windows/macOS

#### M-3. Python engine `_walk` recursion depth equals route length ≤ max_hops (max 8) — safe in practice but no explicit recursion guard
- **Location**: `src/arbcore/engine.py:37-69`
- **Evidence**: Python's default recursion limit is 1000. With `max_hops ≤ 8`, depth is bounded to 8, so stack overflow is not possible. This is accounted for, but there is no explicit comment or assertion documenting that invariant.
- **Impact**: Low; maintainability risk.
- **Severity**: Low

#### M-4. Contract template `FlashArbExecutor.sol` declares `contract` at top level and uses `keccak256` — could be misinterpreted during copy-paste
- **Location**: `contracts/solidity/FlashArbExecutor.sol:7`, `contracts/vyper/FlashArbExecutor.vy`
- **Evidence**: While clearly marked `NOT AUDITED` and `ARBCORE_CONTRACT_TEMPLATE`, the contract has a real `contract` declaration and uses `keccak256`. A developer unfamiliar with the scanner could copy-paste and attempt deployment.
- **Impact**: Misuse risk; governance/release boundary is implicit docs only.
- **Severity**: Medium
- **Likelihood**: Moderate given README scale

#### M-5. CI doesn't run `validate_all.py --include-rust` end-to-end despite running `cargo test`
- **Location**: `.github/workflows/ci.yml:33-36`
- **Evidence**: CI runs `cargo test` and separately runs `scripts/validate_all.py` (last step is `--include-rust` already). Re-reading: line 34 does run `scripts/validate_all.py --include-rust`. **This finding is incorrect upon verification.** The CI does run it. Disregard.

#### M-6. `pyproject.toml` declares entry_points for both `defi-arbitrage-core` and `arbcore`
- **Location**: `pyproject.toml:31-32`
- **Evidence**: Two CLI entry points are declared. `arbore/cli.py:main` and `arbcore/cli.py:main` both map to the same function via the public alias. It's intentional aliasing but adds confusion.
- **Impact**: Branding/cosmetic risk.
- **Severity**: Low

### LOW

#### L-1. `tests/__pycache__` directory appears in tests directory listing
- **Location**: `tests/`
- **Evidence**: `read` of the `tests/` directory listed `__pycache__/` as an entry. `.gitignore` excludes it, but presence in the tree is a hygiene red flag.
- **Impact**: Cosmetic.

#### L-2. `CREATE_NEW_REPO.sh` hard-codes paths (`~/Documents/Google-antigravity/defi-arbitrage-core-mvp-v1`)
- **Location**: `CREATE_NEW_REPO.sh:57`, `scripts/create_new_repo.sh:11,63`
- **Impact**: Limits portability; not a security issue.

#### L-3. `diagnostics_json` includes `sys.executable` absolute path
- **Location**: `src/arbcore/diagnostics.py:26`
- **Impact**: Leaks environment detail; mitigated by docs warning.

#### L-4. Docker base image not pinned by digest
- **Location**: `Dockerfile:1`
- **Impact**: Supply-chain trust reliance on tag mutability.

#### L-5. `validate_rust_service.py` hardcodes `target/debug`
- **Location**: `scripts/validate_rust_service.py:13`
- **Impact**: Fails silently when Rust binary is compiled with `--release`.

## 5. Verification Gaps — Things Cannot Be Confirmed Locally

1. **Old `.git` history contents**: The working tree is sanitized, but the pre-reinit `.git` directory exists outside the repo. Cannot verify whether it contains credentials without external scanning tools.
2. **Python/Rust engine parity for all fixtures**: `scripts/validate_rust_service.py` needs an active `cargo` environment to build and compare; the current workspace has a pre-built `target/` directory but parity cannot be confirmed without execution.
3. **DFS exponential complexity in practice**: Cannot confirm CPU-time behavior for pathological (MAX_EDGE_DEGREE, MAX_HOPS) inputs without running stress tests.
4. **`diagnostics.py` regex and float behavior cross-platform**: Cannot verify serialization output on Windows or macOS without execution there.
5. **Contract template correctness**: Static template validation does not replace compilation or audit; claiming otherwise would be misleading.

## 6. Strong Resume

Overall, the codebase is **well-structured, security-conscious, and appropriately scoped**. The fail-closed validation story is strong: unknown keys are rejected, numerics are bounded, the Rust boundary is isolated behind a process boundary with fallback, and release readiness checks are explicit. The key systemic risks are:

- Residual exposure from unreviewed git history (C-1).
- The Rust engine's looser invariant set vs. Python (H-1, H-2) undermines the "Python is canonical" assurance.
- Star imports (H-3) and absolute-path diagnostics (C-2) are hygiene issues that compound risk.
- CLI silent fallback (M-1) is an operational reliability trap.

The remediation plan prioritizes **verification of the git history first** (because it blocks public release per the project's own `release_readiness.py`), then **Rust parity fixes** (because they affect the integrity of the canonical boundary), then **runtime hardening and diagnostics hygiene**.
