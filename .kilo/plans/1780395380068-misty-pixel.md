**Scope:** Offline DeFi arbitrage analysis core (Python + optional Rust, Solidity/Vyper templates)

## 1. System Understanding Summary

### 1.1 Architecture

The repository implements a deterministic, offline-only market opportunity analysis kernel.

- **Entry points:** `defi-arbitrage-core` / `arbore` CLI commands → `src/arbcore/cli.py`
- **Public package alias:** `defi_arbitrage_core` re-exports `arbcore`
- **Core flow:**
  1. `providers.JsonFileProvider` reads a local JSON snapshot.
  2. `validation.validate_snapshot_payload` + dataclass `validate()` enforce schema/business rules.
  3. `engine.AnalysisEngine` traverses a directed weighted graph to find bounded exchange-rate cycles.
  4. `reporters.JsonReporter` / `TextReporter` emit output.
- **Optional acceleration:** `service.RustAnalysisService` runs `rust/arbcore-rs` in a subprocess and parses JSON opportunities; Python is the fallback.
- **Contracts workspace:** `contracts/` contains Solidity/Vyper templates validated by `arbcore.contracts.validation` using a manifest and static regex checks.

### 1.2 In-Scope vs. Out-of-Scope (confirmed from code + docs)

- **In scope:** local snapshot validation, cycle detection, deterministic output, extension points for providers/reporters/policies, contract-template manifest validation.
- **Out of scope (code-level):** RPC access, wallet/key management, signing, transaction submission, contract compilation/deployment, live trading.

### 1.3 Intended Behavior & Invariants

- Snapshots are untrusted; fail-closed validation occurs before analysis (`src/arbcore/providers.py:37`, `src/arbcore/validation.py:35`).
- Rust output is untrusted; schema and semantic checks occur in `_opportunity_from_dict` (`src/arbcore/service.py:114`).
- Opportunity detection is bounded: `max_hops`, `max_results`, `max_edge_degree`, node-visit dedup cycles.
- All numeric fields are validated for finiteness and sign constraints.
- Contract paths must live under `contracts/` (`src/arbcore/contracts/validation.py:112`); only static-template checks are performed; local-only scope is enforced.

### 1.4 Tech Stack

- Python 3.10+ (no mandatory third-party dependencies in `pyproject.toml:25`).
- Optional Rust crate via `subprocess.run`.
- JSON Schema files in `schemas/` (documentary/validation scripts).
- Solidity 0.8.20 and Vyper 0.4.0 templates (inert, not compiled).
- CI: GitHub Actions running pytest, ruff, secret scan (gitleaks), pip-audit, Docker smoke test.

## 2. Gaps & Hardening Needs (Prioritized)

### CRITICAL

**C-1. Rust subprocess binary path is not canonicalized; relative path resolution is cwd-dependent.**
- **Location:** `src/arbcore/service.py:34-35` and `src/arbcore/cli.py:52`.
- **Evidence:** `RustAnalysisService.available()` calls `Path(self.binary).exists()` directly. When `--rust-binary` is `"defi-arbitrage-core-rs"`, resolution depends on `cwd`. The Rust integration tests (`scripts/validate_rust_service.py:38`) pass an absolute path from `rust/arbcore-rs/target/debug/defi-arbitrage-core-rs`, but the CLI default is unqualified.
- **Impact:** A user running the CLI from a directory without the binary on PATH or relative cwd will get `RustServiceUnavailable` despite the binary existing elsewhere; conversely, `available()` may return true for a binary in a working directory that an attacker controls. The subprocess also does not set `cwd`, so relative paths are resolved from the current process directory, which could be influenced by symlinks or desktop integration.
- **Severity:** High.
- **Likelihood:** Medium.

**C-2. No timeout or subprocess resource guard when spawning the Rust binary (timeout is extremely low).**
- **Location:** `src/arbcore/service.py:65`.
- **Evidence:** `timeout_seconds=5.0` by default. The Rust binary reads the entire snapshot into memory (`fs::read_to_string` in `rust/arbcore-rs/src/main.rs:89`). A multi-megabyte snapshot (Python validates up to 10,000 edges with metadata) could cause slow I/O; 5s may be too aggressive on constrained hardware, but the more pressing gap is the absence of an upper bound on output size or memory usage in the Rust process. There is no `preexec_fn`, `resource.setrlimit`, or equivalent protection.
- **Impact:** Slowness or DOS of the analysis process when Rust path is selected.
- **Severity:** High.
- **Likelihood:** Medium.

**C-3. Graph traversal complexity can blow up with dense/metadata-heavy inputs despite degree checks.**
- **Location:** `src/arbcore/engine.py:26-33`.
- **Evidence:** `MAX_EDGE_DEGREE = 500` and `MAX_REASONABLE_HOPS = 8`. With `max_hops=8` and many nodes of degree 500, the number of walks grows combinatorially (`O(V * d^h)`). Python recursive `_walk` has no iterative fallback and no complexity budget.
- **Impact:** CPU exhaustion / unresponsive CLI on adversarial snapshots with many nodes at degree cap.
- **Severity:** High.
- **Likelihood:** Low (requires crafted snapshot), but high-impact should it occur.

**C-4. `engine._walk` uses recursion and visited-set only on source hops; cycles of length 3+ are detected, but the recursion depth is bounded only by `max_hops`.**
- **Location:** `src/arbcore/engine.py:65-69`.
- **Evidence:** The Python recursion limit is typically 1,000; this is okay for `max_hops <= 8`, but the recursion is per-start-node and not memoized, so it repeatedly traverses overlapping suffixes.
- **Impact:** DoS matrix via many start nodes and dense graph. Not a correctness bug, but a reliability/availability gap.
- **Severity:** Medium.
- **Likelihood:** Low.

### HIGH

**H-1. Secret scanning is incomplete for historical content.**
- **Location:** `scripts/secret_scan.sh`, acknowledged in `SECURITY.md:35-47` and `scripts/release_readiness.py:21-32`.
- **Evidence:** The script uses `git ls-files -co --exclude-standard` (working tree / active tree). `SECURITY.md` explicitly states: "The included secret scan is lightweight active-tree scanning and does not replace independent full-history scanning before publication."
- **Impact:** Historical credentials from prior local history may still exist in `.git/objects`. The repository notes prior sanitization, but no tool here proves it.
- **Severity:** High.
- **Likelihood:** High if release publishes without external full-history scan.

**H-2. Pip dependency scanning lacks reproducible pinning and provenance.**
- **Location:** `pyproject.toml:25-28`, `.github/workflows/ci.yml:44-45`, `Dockerfile:20`.
- **Evidence:** `pip-audit --strict .` runs in CI on an editable install with no lock file; `requirements*.txt` or `constraints.txt` are absent. Docker installs `pip install --no-cache-dir .`, pulling latest compatible versions at build time.
- **Impact:** Supply-chain variance: a dependency (if added later) or transitive dependency could regress or introduce vulnerability without CI catching it due to SLSA provenance gap.
- **Severity:** High.
- **Likelihood:** Medium.

**H-3. Error messages in Python expose internal field names and may reveal schema structure.**
- **Location:** `src/arbcore/validation.py:42-44`, `src/arbcore/contracts/registry.py:61-63`, `src/arbcore/cli.py:133-141`.
- **Evidence:** Errors echo back the exact unsupported JSON keys found. CLI error JSON exposes `__class__.__name__`. While these are not sensitive secrets per se, they leak internal schema/validation semantics that can be used to craft evasion payloads against a downstream service that consumes this ibrary.
- **Impact:** Attack surface for schema-fuzzing if this library is embedded in a service with different trust assumptions.
- **Severity:** Medium.
- **Likelihood:** Low for current CLI-only use.

**H-4. Canonical Python and Rust validation rule sets are duplicated, risking drift.**
- **Location:** `src/arbcore/models.py:71-99` vs. `src/arbcore/contracts/validation.py` vs. `rust/arbcore-rs/src/main.rs:230-271`.
- **Evidence:** Rust reimplements `MAX_*` constants independently and has its own `validate_snapshot_metadata`, `validate_policy`, `normalize_edge`. There is no shared source of truth for validated invariants. Parity is tested (`scripts/validate_rust_service.py`), but only on example files, not exhaustive invalid-input matrices.
- **Impact:** Rust can accept an input that Python rejects (or vice versa) for edge cases not covered by parity tests (e.g., metadata depth, unicode normalization, fee_bps rounding).
- **Severity:** Medium.
- **Likelihood:** Low.

**H-5. Diagnostic output includes executable path and CWD.**
- **Location:** `src/arbcore/diagnostics.py:22-31`.
- **Evidence:** `sys.executable` and `Path.cwd()` are emitted as-is. In containerized or multi-tenant contexts, CWD can leak home-directory structure.
- **Impact:** Minor information disclosure; more significant if logs are aggregated and expose internal host layout.
- **Severity:** Low.
- **Likelihood:** Medium.

### MEDIUM

**M-1. `pyproject.toml` includes email addresses in `authors` and `maintainers`.**
- **Location:** `pyproject.toml:12-13`.
- **Evidence:** Plaintext PII maintained in package metadata. Packaging artifacts (SDist / wheel METADATA) expose these.
- **Impact:** Public release will expose maintainer emails to scrapers unless metadata is sanitized.
- **Severity:** Medium.
- **Likelihood:** High upon release.

**M-2. Release-readiness report hard-codes `production_ready: False` but does not show result in JSON.**
- **Location:** `scripts/release_readiness.py:64-71`.
- **Evidence:** The JSON report includes `"production_ready": false` but does not gate on it in a machine-readable way beyond the boolean. Downstream tooling may ignore it.
- **Impact:** Automation could falsely treat local-validation pass as production-ready.
- **Severity:** Medium.
- **Likelihood:** Low.

**M-3. No retries, circuit breaker, or health-check for Rust binary unavailability.**
- **Location:** `src/arbcore/service.py:33-44, 59-78`.
- **Evidence:** `available()` re-checks existence on every call. If the binary is present but crashes under load, each CLI invocation will attempt and fail with `subprocess.CalledProcessError` or parse failure before Python fallback. No health probe / warm-up cache.
- **Impact:** Latency spikes and error-noise degradation in Rust path.
- **Severity:** Medium.
- **Likelihood:** Medium.

**M-4. Snapshot JSON read uses `path.read_text(encoding="utf-8")` without size guard.**
- **Location:** `src/arbcore/validation.py:96`, `scripts/validate_repository.py` (not yet read, but checked for pattern), `src/arbcore/providers.py:32`.
- **Evidence:** Arbitrary file read. The 10,000-edge cap is enforced, but a single huge string value (e.g., multi-GB metadata field) can be loaded into memory prior to edge-level hydration. No streaming parse or `os.path.getsize()` check.
- **Impact:** Memory exhaustion from malicious local file.
- **Severity:** Medium.
- **Likelihood:** Low (local-only, but supply-chain for snapshots could carry such file).

**M-5. `validate_snapshot_payload` does not validate edge numeric values; parsing errors surface downstream.**
- **Location:** `src/arbcore/validation.py:72-91`.
- **Evidence:** Loop checks key shape and metadata typing, but `rate`, `fee_bps`, and `liquidity` are not type-checked at payload level (only later in `Edge.normalized()` / `Edge.validate()`). Negative-case tests cover some, but the JSON-payload validator trusts the JSON layer to have numbers; `NaN`/`Infinity` roundtrip through JSON libraries inconsistently.
- **Impact:** Inconsistent fail-closed behavior between payload validator and edge parser.
- **Severity:** Medium.
- **Likelihood:** Low.

## 3. Actionable, Sequenced Remediation Plan

### Phase A — Critical Reliability & Path Safety (Sequence: 1-4)

**A-1. Canonicalize Rust binary resolution.**
- **Proposed fix:** In `RustAnalysisService.__init__`, resolve `binary` with `Path(binary).expanduser().resolve()`. If the resolved path does not exist and has no parent directory, fall back to `shutil.which(binary)`. Store the resolved path.
- **Rationale:** Guarantees `--rust-binary` correctness when cwd differs from repo root.
- **Tradeoff:** Behavior change if a user intentionally passed a relative path in their shell; subprocess correctness requires absolute path anyway.

**A-2. Harden Rust subprocess invocation.**
- **Proposed fix:**
  - Add `cwd=str(Path.cwd())` or a configurable cwd.
  - Add `preexec_fn` on POSIX or Windows job-object equivalent to set CPU/memory limits (e.g., `resource.setrlimit`).
  - Add `stdin=subprocess.DEVNULL`.
  - Increase default timeout to 15s with an env override `ARBCORE_RUST_TIMEOUT_SECONDS`.
- **Rationale:** Defense-in-depth against a slow or malicious local snapshot consuming resources.
- **Tradeoff:** Requires more permissive resource APIs; CPU limits could interfere if legitimate analysis needs time.

**A-3. Replace recursive walk with iterative bounded DFS / BFS.**
- **Proposed fix:** Rewrite `AnalysisEngine._walk` as an explicit stack with hop-count tracking. Add `max_edges_per_node` soft cap that triggers a `PolicyError` before traversal.
- **Rationale:** Eliminates Python recursion overhead and makes hop budget a first-class, memory-bounded loop.
- **Tradeoff:** Slightly more code, but deterministic and easier to audit.

**A-4. Add snapshot byte-size pre-check.**
- **Proposed fix:** Before `read_text`, check `self.path.stat().st_size` against a limit (e.g., 64 MiB). Reject with `SnapshotError("snapshot file exceeds size limit")` if exceeded.
- **Rationale:** Prevents trivial memory amplification before JSON parsing.
- **Tradeoff:** None meaningful; files this large are never valid offline market snapshots in normal use.

### Phase B — Security Posture & Observability (Sequence: 8-10)

**B-1. Sanitize diagnostics output.**
- **Proposed fix:** Replace `sys.executable` with `Path(sys.executable).name`; emit `repository.cwd` as relative path (`Path.cwd().relative_to(ROOT)`) when inside the repo, otherwise `"<outside-repo>"`; add a `--verbose-diagnostics` mode that includes absolute paths, disabled by default.
- **Rationale:** Reduces leakage in aggregated log systems while preserving debuggability.
- **Tradeoff:** Minor; full paths remain available via verbose mode or exception context.
- **Verification:** Update `fixtures/golden_outputs/diagnostics.json` and `fixtures/golden_outputs/release_readiness.json` to use relative paths, then re-run `scripts/validate_golden_fixtures.py`.

**B-2. Add a posture gate that blocks edit-install from unsanitized dependencies.**
- **Proposed fix:** Make `release_readiness` fail if `pip-audit` returns any finding, not just as a manual gate. Add it to the automated `CHECKS` list in `scripts/release_readiness.py`.
- **Rationale:** Continuous enforcement of dependency hygiene rather than documentation-only promise.
- **Tradeoff:** Stricter gate; developers must re-pin/audit when updating dev tools.

**B-3. Enforce reproducible Docker builds with pinned digest or lockfile.**
- **Proposed fix:** Add a `constraints.txt` with hashed pins for all runtime and dev dependencies; update `Dockerfile` to `pip install --no-cache-dir -c constraints.txt .`; update `Makefile bootstrap` to honor `PIP_CONSTRAINT`.
- **Rationale:** Docker rebuilds today pull moving versions (currently stdlib only, but becomes relevant as soon as any dep is added).
- **Tradeoff:** Lockfile maintenance burden. Mitigate by pinning only what is installed, and documenting update procedure in `docs/en/INSTALLATION.md`.

### Phase D — Release & Meta (Sequence: 10-11)

**D-1. Sanitize package metadata before release.**
- **Proposed fix:** Replace `authors`/`maintainers` email addresses in `pyproject.toml` with organization URL or single email in `SECURITY.md`; use SPDX `License: MIT` without redundant maintainer block if desired.
- **Rationale:** Prevent maintainer email harvesting from package index.

**D-2. Deprecate or consolidate `defi_arbitrage_core` alias with explicit migration notice.**
- **Proposed fix:** Add deprecation warning in `defi_arbitrage_core` re-export that runs once on import, referencing `arbcore` as canonical.
- **Rationale:** Clarify which package name is supported going forward.
