# نمای کلی تفصیلی

## این زیرساخت برای چه کسانی مناسب است

این زیرساخت برای افرادی و تیم‌هایی ساخته شده که به تحلیل قطعی بازار دیفای نیاز دارند، بدون اینکه ریسک عملیاتی یک سیستم معامله زنده را به ارث ببرند.

برای این گروه‌ها مناسب است:

- researcherهایی که می‌خواهند ایده‌های path-finding را روی snapshotهای محلی بررسی کنند
- توسعه‌دهندگان کمی که می‌خواهند ساختار بازار را میان چند شبکه EVM مقایسه کنند
- تحلیل‌گران پروتکل که topology نقدینگی و کیفیت routeها را مطالعه می‌کنند
- مهندسان backend که سیستم‌های scoring، alerting یا simulation می‌سازند
- شرکت‌هایی که قبل از افزودن ingestion، execution، governance یا compliance خودشان، به یک هسته تحلیل تمیز نیاز دارند

برای کاربری که دنبال یک محصول one-click برای معامله است، مناسب نیست.

## چه مسئله‌ای را حل می‌کند

بسیاری از تیم‌ها کار را با notebookهای پراکنده، scriptهای ad-hoc یا repositoryهای اغراق‌آمیز bot شروع می‌کنند. در آن حالت پاسخ‌دادن به سؤال‌های پایه مهندسی دشوار می‌شود؛ مثل اینکه کدام ورودی‌ها قابل اعتمادند، آیا یک snapshot همیشه خروجی یکسان می‌دهد، شکست validation چگونه نمایش داده می‌شود، و چطور می‌توان سیستم را به شبکه جدید توسعه داد بدون بازنویسی engine.

این repository همین مسئله محدود اما مهم را با یک هسته reusable و contract-driven حل می‌کند.

## مدل استفاده عملی

مدل واقع‌بینانه این است که این هسته در میانه یک pipeline بزرگ‌تر قرار بگیرد.

سیستم‌های upstream می‌توانند quote جمع کنند، venueها را نرمال‌سازی کنند، symbolها را enrich کنند یا شرایط را شبیه‌سازی کنند. سپس این هسته snapshot را validation می‌کند، تحلیل چرخه محدود را اجرا می‌کند و candidateهای ساخت‌یافته تولید می‌کند. سیستم‌های downstream هم می‌توانند آن‌ها را score کنند، ذخیره کنند، رویشان alert بسازند، review کنند یا تحت کنترل‌های جداگانه اجرا کنند.

این مدل برای افراد، تیم‌های تحقیقاتی، گروه‌های tooling داخلی و شرکت‌های زیرساختی مناسب است؛ چون هسته را قابل‌فهم نگه می‌دارد و اجازه می‌دهد لایه‌های اطراف مستقل رشد کنند.

## چرا طراحی چندشبکه‌ای مهم است

فیلد صریح `network` باعث می‌شود engine فقط به یک chain محدود نباشد. یک تیم می‌تواند همین هسته قطعی را برای Base، Ethereum، Arbitrum، Optimism، Polygon، Avalanche، BNB Chain، simulationهای محلی یا محیط‌های خصوصی استفاده کند؛ به شرط آنکه snapshotها از همان contract پیروی کنند.

این طراحی reuse، reviewability و test coverage را بهتر می‌کند و جلوی fork شدن بی‌قاعده engine برای هر شبکه را می‌گیرد.

## چرا validation در مرکز است

validation یکی از حرفه‌ای‌ترین بخش‌های این repository است.

validation در سطح schema، شکل عمومی snapshotها، reportها و diagnostics را مستند می‌کند. validation در runtime هم ruleهای کسب‌وکاری مثل وجود صریح network، edgeهای سالم، policyهای محدود و رد fail-closed برای ورودی خراب را enforce می‌کند. کنار هم، این دو باعث می‌شوند سیستم برای embedding در ابزارهای بزرگ‌تر امن‌تر و قابل‌اعتمادتر شود.

## تیم‌ها هنوز چه چیزهایی باید اضافه کنند

این repository عمداً محصول کامل نیست. تیم‌هایی که از آن استفاده می‌کنند هنوز باید درباره این موارد تصمیم بگیرند:

- جمع‌آوری و نرمال‌سازی داده
- storage و retention
- scheduling و orchestration
- risk approval و execution control
- observability، incident response و compliance review
- مدیریت secretها و network access در محیط production

این جداسازی یک نقطه قوت طراحی است، نه کمبود قابلیت.
